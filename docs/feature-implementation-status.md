## Project Scope and Assumptions from PDF

- Geographic coverage focuses on major global tourist hubs, such as the top 100 eco-friendly cities, rather than exhaustive global coverage for every remote location.
- The system is a planner and directory, not a booking engine. It may link to external booking sites, but it does not process hotel or flight transactions.
- The initial release supports English only and targets a responsive web interface optimized for desktop and mobile browsers.
- Carbon calculations assume users provide accurate travel habit data, such as car engine type or flight class.
- Eco-friendly labels from third-party directories or service providers are assumed to be valid; the system does not perform independent on-site audits.
- The application is online-first and assumes stable internet access for directory updates and weather information.

## Additional Implemented Capabilities Found

These are not separate rows in the PDF feature table, but they support or extend the documented scope:

- Eco route planning is implemented through `frontend/src/components/eco-route-planner-section.tsx` and `backend/src/routes/eco-route.ts`, including geocoding, current-location support, walking/cycling/driving route comparisons, OpenRouteService ETAs, and CO2 estimates.
- Directory details and related place recommendations are implemented through `GET /api/locations/:publicId` and `GET /api/locations/recommendations`.
- Carbon tracking history is implemented through `GET /api/carbon/history`, `GET /api/carbon/summary`, `DELETE /api/carbon/entries/:id`, `frontend/src/app/carbonFootprint/history/page.tsx`, and `frontend/src/app/dashboard/carbonHistory/page.tsx`.
- Analytics and reporting are implemented through `GET /api/analytics/impact`, `GET/PUT /api/analytics/goal`, `frontend/src/app/dashboard/analytics/page.tsx`, and the PostgreSQL `carbon_budget_goals` table.
- Admin pages exist for dashboard, destinations, users, and analytics under `frontend/src/app/admin`, protected by role-aware middleware.

## Main Gaps

- Most user-specific features now have PostgreSQL-backed persistence for authenticated users, including profile data, favourites, trips, todos, carbon calculation records, community reviews, and challenge progress.
- Analytics and reporting now use authenticated PostgreSQL-backed carbon calculation and budget goal records rather than hard-coded dashboard data.
